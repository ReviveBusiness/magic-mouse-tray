## Phase 9–11 — Day-of empirical work (2026-04-29 daytime)

The morning install reached the device but **scroll never worked on v3**. The afternoon was spent gathering empirical evidence on why, and the result is an architectural blocker on the M12 design as currently shipped. Full analysis in `M12-EMPIRICAL-BLOCKER-2026-04-29.md`. This section is the timeline.

### Phase 9 — Trace ring buffer + observation-only mode

After v3 cursor worked but scroll did not, the first hypothesis was "we're rewriting the wrong frames." Added a kernel-side trace ring buffer (16 slots × 16 bytes per frame, WDFTIMER 1Hz → WDFWORKITEM → `ZwSetValueKey` to a registry value the user-mode tray reads).

**Empirical finding**: every interrupt frame on RID=0x12 is exactly **9 bytes**, format `A1 12 ...` (transaction header + RID + 7-byte multi-touch payload). v3 firmware emits **only** RID=0x12 multi-touch frames. There is no RID=0x02 traffic at all unless a SET_REPORT enables it.

This invalidated the v8 design's assumption that RID=0x02 mouse-with-Wheel frames would arrive natively. They do not. Disabled the rewrite path with `#if 0` to put the driver in pure observation mode.

### Phase 10 — BTHDDI SET_REPORT injection attempt

If the device emits only RID=0x12 by default, the only way to get scroll is to send a SET_REPORT enabling multi-touch mode — which is exactly what Linux's `magicmouse_enable_multitouch()` does (`{0xF1, 0x02, 0x01}` for PID 0x0323).

Implemented kernel-side BTHDDI injection:
- Added `WdfFdoQueryForInterface` for `GUID_BTHDDI_PROFILE_DRIVER_INTERFACE_LOCAL` ({94A59AA8-…})
- Captured `RemoteBtAddr` from `BRB_OPEN_L2CAP_CHANNEL` at offset 0x70
- On first `OpenChannelCompletion`, called `BthAllocateBrb(BRB_HID_SEND_CONTROL_TRANSFER)`, populated SET_REPORT with feature report 0xF1 payload `{0xF1, 0x02, 0x01}`, submitted via `IoCallDriver` to `WdfDeviceGetIoTarget` with custom completion routine

**Empirical result**: BRB submitted, completion routine fired, status = **`STATUS_INVALID_PARAMETER` (0xC000000D)**. The IRP plumbing reached BthEnum but BthEnum rejected the BRB.

Confirmed via trace: `MtStage=6` (submit attempted) + `MtCompletionStatus=0xC000000D`.

### Phase 11 — NotebookLM corpus query + binary reverse engineering

User: *"don't assume anything. ensure this is backed with empirical data"* and *"we already have the other drivers to work from."*

Two parallel evidence-gathering tracks:

**Track A — `xxd` on Apple's binary + MagicUtilities binary**

Searched both `applewirelessmouse.sys` and `MagicMouse.sys` (MagicUtilities) for the byte sequence `F1 02 01`.

**Result: byte sequence is absent from both binaries.** Apple's filter does not send Linux's feature report `0xF1`, and MagicUtilities does not either. The Linux `magicmouse_enable_multitouch()` mechanism does not transfer to Windows BT.

**Track B — NotebookLM cross-corpus query** (Apple driver + MagicUtilities + Linux hid-magicmouse)

Cross-referenced what each driver actually does to enable scroll on Magic Mouse 2024 BT:

- **Apple `applewirelessmouse.sys`**: sends NO command. v3 firmware *natively* emits RID=0x02 frames (8-bit X / Y / AC-Pan / Wheel) once it sees Apple's HID descriptor. Apple's filter only **replaces the SDP HID descriptor at pairing time** with a 116-byte "Descriptor B" that maps RID=0x02 to standard Mouse usages with embedded AC-Pan + Wheel.
- **MagicUtilities `MagicMouse.sys`**: enables RID=0x12 multi-touch via Feature Report **0x55** (NOT 0xF1) sent from a userland service via `HidD_SetFeature`. Out of scope for M12 (kernel-only filter).
- **Linux**: uses 0xF1 because Linux's BT path lets a HID feature report through; Windows BT does not propagate that path the same way.

Two specialist agents (Agent A: descriptor reverse-engineering of `applewirelessmouse.sys` at offset 0xA850; Agent B: v9 descriptor design) produced `M12-DESCRIPTOR-B-ANALYSIS-2026-04-29.md` and `M12-V9-DESIGN-2026-04-29.md`. The 116-byte HID descriptor at offset 0xA850 declares one Mouse TLC at RID=0x02 with 5 buttons + INT8 X/Y + INT8 AC-Pan (Consumer 0x0238) + INT8 Wheel embedded inside the Mouse TLC.

### Phase 11b — v9 implementation (descriptor injection only)

Rewrote `HidDescriptor.c` to mirror Apple's binary exactly — 2 TLCs:
- TLC1: Mouse, RID=0x02, 5 buttons + X/Y/AC-Pan/Wheel (matches Apple offset 0xA850)
- TLC2: Vendor battery, RID=0x90 (preserved for tray-app battery readout)

Built clean, signed (legacy-CSP via SYSTEM-context queue), installed, **cleared `BTHPORT\Parameters\Devices\<MAC>\CachedServices` registry key**, removed and re-paired the mouse.

**Empirical result — the architectural blocker**:

1. After the re-pair, the BTHPORT cache repopulated with the **same 351-byte SDP service record containing Apple's RID=0x12 descriptor at offset 176** — NOT our 116-byte injected descriptor.

2. Trace ring buffer captured **zero SDP-shaped frames** during pairing. The buffer showed only 9-byte `A1 12 …` HID interrupt frames during cursor motion, which is exactly what we'd see without M12 in place.

3. The "successful" COL02 vendor battery enumeration we kept seeing under every M12 build was actually **Apple's native descriptor** — Apple's stock SDP record already declares RID=0x90 vendor at offset 261 with the right usages. It looked like our injection was working because the device declares the same TLC natively.

So: **our descriptor injection at the BRB lower-filter layer never actually intercepted the SDP HIDDescriptorList exchange.** SDP traffic flows through a path our M12 lower filter does not see — likely a BTHPORT-internal fast-path or a higher-layer query that doesn't propagate as `BRB_L2CA_ACL_TRANSFER` IRPs.

### Phase 11c — Final rollback to Apple

Per user: *"just fix it. I have working mouse!!!"* — rolled back to Apple's `applewirelessmouse` driver via `mm-task-runner.ps1 ROLLBACK-M12`. Both v1 (PID 0x030D) and v3 (PID 0x0323) currently bound to Apple's stock filter; cursor + scroll **working** on v1 (always was), cursor working on v3 (Apple native), scroll not working on v3 (Apple's filter requires Apple's userland context that we don't have).
