# M12 v9 HID Descriptor + Driver Behavior Design

**Date**: 2026-04-29  
**Status**: Design — pending approval before any code change  
**Context**: Parallel agent extracting Apple's exact 116-byte descriptor; that analysis not yet available — this design is built from empirical tray logs, SDP cache decode, DESCRIPTOR-A-vs-B-DIFF.md, SESSION-12-MU-INSTALLED, and M12-V16-EMPIRICAL-LAYOUT-CORRECTION.md.

---

## Mission summary

Tonight's key empirical findings: v3 firmware natively emits RID=0x02 scroll on the interrupt channel. Apple's SDP-injection approach replaces attribute 0x0206 with a descriptor that maps RID=0x02 bytes to standard scroll/cursor. M12 must do the same — inject a descriptor that declares both a scroll-capable mouse TLC (consuming RID=0x02 reports the firmware already emits) AND the vendor battery TLC (RID=0x90, UP:0xFF00/U:0x0014) that Apple's filter silently destroys in its unified mode.

The critical constraint: **the firmware natively emits RID=0x02**. We do NOT need to translate or rewrite reports. We need the descriptor to declare what RID=0x02 means.

---

## 1. New `g_HidDescriptor[]` byte array

**Design baseline**: Apple's Descriptor B (the SDP-injected one) was 116 bytes and declared RID=0x02 as Mouse with X+Y+AC-Pan+Wheel. We know from SESSION-12 that Magic Utilities Mode A declares RID=0x02 with 5 buttons, 8-bit X/Y, 16-bit Wheel, 16-bit AC Pan, plus Resolution Multiplier features at RID=0x03 and RID=0x04.

**Target layout**: Two TLCs. Drop RID=0x01 (was our synthetic mouse; now replaced by RID=0x02 which firmware actually emits). Replace the prior RID=0x02 Consumer TLC (AC Pan only, 1 byte) with a full Mouse TLC consuming the firmware's native RID=0x02 frames. Retain RID=0x90 vendor battery TLC unchanged.

The byte array below mirrors the MU Mode A layout (empirically confirmed working for scroll on v3) minus the Resolution Multiplier features (which add complexity and are not required for basic scroll — they enable high-resolution precision scroll; can add in v10 if needed). This keeps the total under 127 bytes, which avoids SDP length-encoding complications (see Section 3).

```c
// SPDX-License-Identifier: MIT
//
// g_HidDescriptor — M12 v9
//
// Two TLCs injected via SDP attribute 0x0206:
//
//   TLC1 (RID=0x02) — Generic Desktop Mouse
//     Maps the firmware's native RID=0x02 interrupt reports:
//       buf[0] = 0xA1 (HID-BT transaction header, NOT part of report)
//       buf[1] = 0x02 (Report ID)
//       buf[2] = buttons byte (bits 0-4 = buttons 1-5, bits 5-7 = pad)
//       buf[3] = X signed 8-bit
//       buf[4] = Y signed 8-bit
//       buf[5] = AC Pan (horizontal scroll) signed 8-bit
//       buf[6] = Wheel (vertical scroll) signed 8-bit
//     Total wire bytes: 5 data bytes after RID byte = 5-byte report.
//     NOTE: Apple Descriptor B declares 8-bit fields per empirical MODE B
//     evidence (InLen=8 under Descriptor A, X/Y/Wheel/ACPan all 8-bit).
//     InputReportByteLength = 6 (RID + buttons + X + Y + ACPan + Wheel).
//
//   TLC2 (RID=0x90) — Vendor Battery (UP:0xFF00 / Usage:0x0014)
//     Unchanged from current HidDescriptor.c TLC3.
//     Firmware pushes [0x90, flags, pct] unsolicited on interrupt channel.
//     Tray reads via HidD_GetInputReport(0x90) on COL02; buf[2] = pct.
//     InputReportByteLength = 3.
//
// REMOVED vs current HidDescriptor.c:
//   - TLC1 (RID=0x01 Mouse) — no firmware support; mouhid opened exclusive
//     anyway and would conflict with RID=0x02 being in a separate TLC.
//   - TLC2 (RID=0x02 Consumer Control AC Pan only) — replaced by full
//     Mouse TLC below which includes AC Pan within the Mouse collection.
//     Both used RID=0x02 — they CANNOT coexist. One must go. The Consumer
//     version had only 1 byte (AC Pan); the Mouse version carries all fields
//     the firmware emits. Mouse version wins.
//
// Total size: 63 bytes. Well under 127 — 1-byte SDP length encoding safe.

const UCHAR g_HidDescriptor[] = {
    // ----------------------------------------------------------------
    // TLC1: Generic Desktop Mouse (Usage Page 0x01, Usage 0x02)
    // Report ID 0x02 | InputReportByteLength = 6
    // Maps firmware-native RID=0x02 interrupt reports directly.
    // ----------------------------------------------------------------
    0x05, 0x01,       // Usage Page (Generic Desktop)           [parser: set UP=0x01]
    0x09, 0x02,       // Usage (Mouse)                          [parser: TLC usage]
    0xA1, 0x01,       // Collection (Application)              [parser: open App coll]
    0x85, 0x02,       //   Report ID (2)                       [all items below → RID=0x02]

    0x09, 0x01,       //   Usage (Pointer)
    0xA1, 0x00,       //   Collection (Physical)               [parser: open Phys coll]

    // Buttons: 5 × 1-bit + 3-bit pad = 1 byte (buf[2] of firmware report)
    0x05, 0x09,       //     Usage Page (Button)
    0x19, 0x01,       //     Usage Minimum (Button 1 — left click)
    0x29, 0x05,       //     Usage Maximum (Button 5)          [matches MU Mode A: 5 buttons]
    0x15, 0x00,       //     Logical Minimum (0)
    0x25, 0x01,       //     Logical Maximum (1)
    0x75, 0x01,       //     Report Size (1 bit)
    0x95, 0x05,       //     Report Count (5)
    0x81, 0x02,       //     Input (Data, Variable, Absolute)  [5 button bits → buf[2] bits 0-4]
    0x75, 0x03,       //     Report Size (3 bits — padding)
    0x95, 0x01,       //     Report Count (1)
    0x81, 0x03,       //     Input (Constant)                  [3 pad bits → buf[2] bits 5-7]

    // X axis — INT8 relative (buf[3])
    0x05, 0x01,       //     Usage Page (Generic Desktop)
    0x09, 0x30,       //     Usage (X)
    0x15, 0x81,       //     Logical Minimum (-127)
    0x25, 0x7F,       //     Logical Maximum (127)
    0x75, 0x08,       //     Report Size (8)
    0x95, 0x01,       //     Report Count (1)
    0x81, 0x06,       //     Input (Data, Variable, Relative)  [X → buf[3]]

    // Y axis — INT8 relative (buf[4])
    0x09, 0x31,       //     Usage (Y)
    0x15, 0x81,       //     Logical Minimum (-127)
    0x25, 0x7F,       //     Logical Maximum (127)
    0x75, 0x08,       //     Report Size (8)
    0x95, 0x01,       //     Report Count (1)
    0x81, 0x06,       //     Input (Data, Variable, Relative)  [Y → buf[4]]

    // AC Pan — Consumer UP 0x0C, Usage 0x238, INT8 (buf[5] = horizontal scroll)
    // MUST be declared inside the Mouse TLC here (not a separate Consumer TLC)
    // because mouhid holds the Mouse TLC exclusive — a separate Consumer TLC
    // opened shared would never receive RID=0x02 bytes from the same report.
    // Apple Descriptor B embeds AC Pan inside the Mouse TLC for this reason.
    0x05, 0x0C,       //     Usage Page (Consumer Devices)
    0x0A, 0x38, 0x02, //     Usage (AC Pan 0x0238)            [2-byte extended usage LE]
    0x15, 0x81,       //     Logical Minimum (-127)
    0x25, 0x7F,       //     Logical Maximum (127)
    0x75, 0x08,       //     Report Size (8)
    0x95, 0x01,       //     Report Count (1)
    0x81, 0x06,       //     Input (Data, Variable, Relative)  [ACPan → buf[5]]

    // Vertical scroll wheel — Generic Desktop Wheel 0x38, INT8 (buf[6])
    0x05, 0x01,       //     Usage Page (Generic Desktop)
    0x09, 0x38,       //     Usage (Wheel)
    0x15, 0x81,       //     Logical Minimum (-127)
    0x25, 0x7F,       //     Logical Maximum (127)
    0x75, 0x08,       //     Report Size (8)
    0x95, 0x01,       //     Report Count (1)
    0x81, 0x06,       //     Input (Data, Variable, Relative)  [Wheel → buf[6]]

    0xC0,             //   End Collection (Physical)
    0xC0,             // End Collection (Application)

    // ----------------------------------------------------------------
    // TLC2: Vendor Battery (Usage Page 0xFF00, Usage 0x14)
    // Report ID 0x90 | InputReportByteLength = 3
    // Firmware pushes [0x90, flags, pct] on interrupt channel.
    // Tray reads buf[2] via HidD_GetInputReport on COL02.
    // ----------------------------------------------------------------
    0x06, 0x00, 0xFF, // Usage Page (Vendor-Defined 0xFF00)
    0x09, 0x14,       // Usage (0x14)
    0xA1, 0x01,       // Collection (Application)
    0x85, 0x90,       //   Report ID (0x90)
    0x09, 0x01,       //   Usage (0x01) — flags byte             [buf[1]]
    0x09, 0x02,       //   Usage (0x02) — battery percent byte   [buf[2]]
    0x15, 0x00,       //   Logical Minimum (0)
    0x26, 0xFF, 0x00, //   Logical Maximum (255)
    0x75, 0x08,       //   Report Size (8)
    0x95, 0x02,       //   Report Count (2)
    0x81, 0x02,       //   Input (Data, Variable, Absolute)
    0xC0,             // End Collection
};

const ULONG g_HidDescriptorSize = sizeof(g_HidDescriptor);
// Expected: 63 bytes. Verify with static_assert(sizeof(g_HidDescriptor) == 63, "");
```

**Wire-byte mapping summary** (firmware interrupt report after 0xA1 transaction header):

| Byte | Descriptor field | HID usage |
|------|-----------------|-----------|
| buf[1] | Report ID | 0x02 |
| buf[2] | Buttons 1-5 + 3 pad bits | UP=0x09 Usage 1-5 |
| buf[3] | X | GP Desktop 0x30 |
| buf[4] | Y | GP Desktop 0x31 |
| buf[5] | AC Pan (H-scroll) | Consumer 0x0238 |
| buf[6] | Wheel (V-scroll) | GP Desktop 0x38 |

## 2. Design decisions

**REMOVE RID=0x01 TLC (current TLC1).**  
The firmware does not emit RID=0x01 reports. The old TLC1 was written for a synthetic translation approach that was later abandoned. Keeping it creates a phantom HID interface that mouhid would open exclusive on a report ID no data ever arrives for. More critically, it forces HidClass to enumerate a COL01 interface for RID=0x01 AND a separate one for RID=0x02 — the firmware's actual reports hit the COL for RID=0x02 while mouhid binds to COL01 (exclusive open). Cursor + scroll would both break. Remove.

**REPLACE (not keep) the current TLC2 (RID=0x02 Consumer Control).**  
TLC2 currently declares RID=0x02 as Consumer Control with only AC Pan. The firmware emits RID=0x02 with all fields (buttons, X, Y, AC Pan, Wheel). The new TLC1 (index position in the byte array) claims RID=0x02 as Mouse with all fields. Two TLCs sharing RID=0x02 would cause HID parser undefined behavior — the parser uses RID to route fields and cannot resolve a conflict. The Consumer TLC is removed and AC Pan is embedded within the new Mouse TLC. This is what Apple's Descriptor B does, and it's architecturally correct: since mouhid opens the Mouse TLC exclusive by `HidP_GetCollectionCapabilities`, no other driver opens it, and AC Pan embedded within the Mouse collection is fully delivered to mouhid which generates `WM_MOUSEHWHEEL`.

**INF: no change needed.** Existing INF targets `BTHENUM\...PID&0323` and installs M12 as a lower filter regardless of descriptor content. Descriptor is injected dynamically via SDP; INF only governs filter attachment.

## 3. SDP injection — PatchSdpHidDescriptor correctness analysis

Current implementation pattern in `InputHandler.c`:

```
descOffset-7 : outer SEQUENCE 1-byte length (0x35 LL)
descOffset-5 : inner SEQUENCE 1-byte length (0x35 LL)  
descOffset-1 : TEXT_STRING 1-byte length (0x25 NN)
descOffset+0 : <NN bytes of HID descriptor>
```

**1-byte length form capacity**: `0x35` = SEQUENCE type with 1-byte length, max payload = 255 bytes.

**For v9 descriptor (63 bytes)**:
- `newDescLen` = 63 ✓ (< 255, `if (newDescLen > 0xFF) return FALSE` doesn't fire)
- `innerPayload` = 2 + 2 + 63 = 67 ✓ (< 255)
- `outerPayload` = 2 + 67 = 69 ✓ (< 255)

All three length bytes fit in UCHAR. **No changes to `PatchSdpHidDescriptor` are required.**

**The 2-byte length encoding concern**: If a future descriptor exceeds 127 bytes (where SDP might use `0x36` 2-byte form), the current scanner returns FALSE at `buf[i+3] != SDP_DE_SEQUENCE_1B` — safe degradation, no patch applied. At 63 bytes we are nowhere near this boundary. The current code has explicit overflow guards and will handle this correctly without modification.

**Buffer size constraint**: `PatchSdpHidDescriptor` checks `newBufSize > bufSize` and bails if the patched content won't fit in the original buffer. Since the new descriptor (63 bytes) is smaller than the current descriptor (111 bytes), the new buffer used will be 48 bytes SMALLER than before. No overflow possible.

**The `#if 0` block in InputHandler.c** (report translator, observation-only mode): Leave it in place — it doesn't execute, has no runtime cost. If RID=0x02 native frames start arriving post-redesign and we need to confirm byte layout, the trace ring will capture them. No need to enable the translator because descriptor injection is the mechanism.

**The `InputHandler_SendMultitouchEnable` call**: Per mission constraint #2, the Linux SET_REPORT(0xF1, 0x02, 0x01) approach does not apply on Windows BT. However, the code path is still compiled in and fires on first OPEN_CHANNEL completion. This will send a 4-byte ACL OUT transfer to the device; the device will either silently ignore it or NACK. Either way it is not harmful. The mission says this does NOT need to happen — but it also does no damage. Leave it. If the trace ring shows an NACK disturbing the pairing exchange, disable via a DevCtx flag, but don't preemptively remove working code.

## 4. Expected behavior post v9 install (first re-pair)

1. User unpairs v3 mouse, re-pairs.
2. BT stack performs SDP attribute exchange on PSM 1 (L2CAP control channel).
3. Device publishes its native SDP HIDDescriptorList (attribute 0x0206) — this is the multi-TLC descriptor from the SDP cache: RID=0x12 Mouse + Feature 0x55 (touchpad-mode vendor) + RID=0x90 battery.
4. M12's `AclCompletion` fires on the incoming ACL transfer carrying the SDP response. `ScanForSdpHidDescriptor` finds the `09 02 06 35 LL 35 LL 08 22 25 NN` pattern.
5. `PatchSdpHidDescriptor` overwrites the embedded descriptor bytes with `g_HidDescriptor` (63 bytes). Length bytes at `descOffset-1/-5/-7` updated to 63/67/69 respectively.
6. HidBth caches the injected 63-byte descriptor. At enumeration time HidClass calls `HidP_GetCollectionCapabilities` on the cached descriptor.
7. HidClass finds TWO Application TLCs: RID=0x02 Mouse + RID=0x90 Vendor. Enumerates two child PDOs:
   - `COL01`: Mouse TLC — mouhid binds, creates `HID\VID_..._PID_0323&COL01`
   - `COL02`: Vendor battery TLC — HidClass creates `HID\VID_..._PID_0323&COL02` with UP:FF00/U:0014
8. Firmware begins emitting RID=0x02 interrupt reports (it natively does this per mission finding #1). mouhid receives them on COL01, maps fields per our descriptor → cursor moves, scroll generates `WM_MOUSEWHEEL` and `WM_MOUSEHWHEEL`.
9. Firmware pushes RID=0x90 battery reports periodically. Tray opens COL02 by UP:FF00/U:0014, calls `HidD_GetInputReport(0x90)`, gets `[0x90, flags, pct]`, returns `buf[2]`.
10. Both battery and scroll work simultaneously — the problem neither Apple's filter nor Magic Utilities trial state could achieve.

## 5. Risks and unknowns

**R1 — Is the firmware's RID=0x02 byte layout actually `[buttons, X8, Y8, ACPan8, Wheel8]`?**  
HIGH RISK. This is the central assumption. Evidence supporting it: Apple's Descriptor B declares exactly these fields at INT8 each, and Apple built this descriptor by capturing and mapping the actual device output. The mission finding that "v3 firmware natively emits scroll on RID=0x02 with 8-bit X/Y/AC-Pan/Wheel fields" is stated empirically. However, this was inferred from Descriptor B's structure and the MU Mode B tray evidence — we have NOT directly observed a raw RID=0x02 frame from this device on Windows (the trace ring showed RID=0x12, not RID=0x02, because Apple's filter was not on the stack). **Validation plan**: After installing v9, read the trace ring from the registry key `Services\MagicMouseDriver\Diag\RecentFrames`. If RID=0x02 frames appear with the expected layout, confirmed. If RID=0x12 frames still appear (but no RID=0x02), the descriptor is wrong — the firmware's native format may require a different mapping or the SET_REPORT enable IS needed after all.

**R2 — Does HidBth re-fetch SDP on every reconnect or only on first pair?**  
MEDIUM RISK. If HidBth caches the descriptor in the registry (PnP device parameters key) and does NOT re-fetch on reconnect, then M12 only gets one injection opportunity per pair event. Subsequent reconnects (sleep/wake, Bluetooth range loss) use the cached descriptor — our injected version. This is the DESIRED behavior: inject once, works forever until user unpairs. But if HidBth re-fetches on every reconnect and M12 is not on the stack for the reconnect (e.g., filter detached), we'd lose the injection. The empirical evidence from PHASE-E-FINDINGS.md (DSM property write triggered a descriptor flip) suggests HidBth CAN re-fetch — but this was under a different driver. Mitigation: the trace ring will show SDP patterns on reconnect; if we see them, injection fires again automatically.

**R3 — Do TWO TLCs in the injected descriptor work, or does Apple use just one?**  
LOW-MEDIUM RISK. Apple's Descriptor B apparently had one TLC (the Mouse one) based on the fact that Apple's filter in Mode B produces a single combined interface. However, Mission finding #3 says Apple replaces 0x0206 with a 116-byte Descriptor B that declares "RID=0x02 as Generic Desktop Mouse with X+Y+AC-Pan+Wheel" — it does not say it ALSO declares the vendor battery TLC. If Apple's injected descriptor is single-TLC only, then the device's native multi-TLC SDP descriptor gets replaced with a simpler single-TLC version, and the vendor battery TLC is simply absent (explaining why Mode B has no COL02). M12's TWO-TLC approach is architecturally different from Apple's — it keeps the vendor battery TLC alive. The risk is that some Windows HidBth behavior rejects or truncates a multi-TLC descriptor injected via SDP; this has not been empirically tested. Mitigation: the parallel agent's Descriptor B analysis (pending in `/tmp/m12-descriptor-b-analysis.md`) will confirm or deny this. If Descriptor B is truly single-TLC, we may need to add the battery TLC as an ADDITION to whatever Apple's descriptor contains rather than designing from scratch.

**R4 — AC Pan inside a Mouse TLC: does mouhid deliver WM_MOUSEHWHEEL?**  
LOW RISK. mouhid on Windows 10/11 handles `Usage=0x238` (AC Pan) within a Mouse TLC and generates `WM_MOUSEHWHEEL`. MU Mode A empirically used 16-bit Wheel/Pan and confirmed scroll worked (when licensed). Our 8-bit version follows the same structure. The Consumer TLC approach in the current HidDescriptor.c (separate RID=0x02 Consumer TLC) was theoretically correct but may have required a third-party Consumer driver to open it — mouhid doesn't open Consumer TLCs. Embedding AC Pan in the Mouse TLC is the right call and matches Apple's approach.

**R5 — Descriptor B parallel agent findings may invalidate byte layout assumptions.**  
If the agent's analysis shows Apple's 116-byte descriptor uses different field widths (e.g., 12-bit or 16-bit for wheel/pan) or a different button count, v9 descriptor bytes will need adjustment before v9 can be considered validated. **Do not cut v9 INF/driver before the Descriptor B report arrives.**
