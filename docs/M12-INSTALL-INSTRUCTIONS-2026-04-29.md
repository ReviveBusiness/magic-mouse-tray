## Pre-flight (read before running anything)

### Why this is risky
M12 is a KMDF lower filter on the BTHENUM stack. A bug in the IRP completion path can BSOD the system. **Have your Dell USB mouse plugged in before installing** (recovery cursor when Magic Mouse v3 is broken).

### What's been verified
- ✅ Compiles cleanly (Release/x64, EWDK 10.0.26100.0) — 4 builds across the night
- ✅ 4-reviewer adversarial code review converged to unanimous APPROVE
- ✅ check-style.sh: 4 pre-existing baseline REJECTs (comment density on header files only — no code defects)
- ✅ Catalog generated by Inf2Cat (signability test PASS, 0 errors, 0 warnings)
- ✅ `MagicMouseFix` cert in `LocalMachine\TrustedPublisher` (driver loads without testsigning watermark)
- ⚠ **Signatures NOT yet applied** — requires user-elevated PowerShell (legacy CSP private key blocks non-elevated access). The morning-deploy script handles this in one step.

### What has NOT been verified
- ❌ **Driver Verifier 0x49bb runtime check** — requires the driver to be installed; can't run statically
- ❌ Behavior on a real Magic Mouse v3 — first install IS the test
- ❌ Recursive TLV SDP parser — current code uses byte-pattern matching (deferred — works for current 113-byte descriptor pattern but fragile if Apple's descriptor changes shape)
- ❌ Channel race resilience under heavy BT load (tested only by code review)
- ❌ PREfast static analysis — vcxproj missing `<CodeAnalysisRuleSet>`, NativeCodeAnalysis CA0001 NullRef. Documented as known tooling gap; the 4-reviewer panel covered the BSOD-likely paths PREfast normally catches (IRP completion, BRB length, IRQL, SAL drift).

## Morning step-by-step

### 1. Open elevated PowerShell

Right-click PowerShell in Start Menu → **Run as Administrator**. Click Yes on UAC.

### 2. Run the deploy script

```powershell
D:\mm3-driver\m12-build-final\m12-morning-deploy.ps1
```

The script will:
1. Verify you're elevated (exits if not)
2. Verify the Apple INF backup exists at `D:\Backups\AppleWirelessMouse-RECOVERY\`
3. Verify `CN=MagicMouseFix` cert is in `LocalMachine\My` with private key
4. Sign `.sys` and `.cat` via signtool with timestamp
5. Verify both signatures via `signtool verify /pa /v`
6. Show staged files + their sizes
7. **Prompt before install** — type `yes` to proceed, anything else to stop here
8. Remove the existing Apple INF from BTHENUM
9. `pnputil /add-driver MagicMouseDriver.inf /install`

If you want to **sign-only** (defer install to manual `pnputil`):
```powershell
D:\mm3-driver\m12-build-final\m12-morning-deploy.ps1 -SignOnly
```

If you want to **skip the install confirmation prompt**:
```powershell
D:\mm3-driver\m12-build-final\m12-morning-deploy.ps1 -YesInstall
```

### 3. Re-pair Magic Mouse v3

- Bluetooth settings → remove Magic Mouse if present → put mouse in pairing mode → re-pair
- This forces BTHENUM to re-enumerate so M12 binds as lower filter on the fresh PDO

### 4. Smoke test

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File D:\mm3-driver\m12-build-final\m12-post-install-smoke-test.ps1
```

The smoke test checks:
- M12 service is registered + Running
- Magic Mouse v3 device is bound to MagicMouseDriver
- HID stack has Report ID 0x90 (vendor battery) — read 3 bytes via `HidD_GetInputReport`
- Tray app log shows recent battery readings

Exit codes: `0` = pass, `2` = degraded, `3` = not installed.

### 5. Functional check

- ✅ Cursor moves
- ✅ Scroll works
- ✅ Tray app shows battery percentage (within 60s of pairing — tray polls)
- ✅ No event log errors at `Windows Logs/System` filtered by source `Microsoft-Windows-Kernel-PnP`

## Known follow-ups (not blocking install)

1. **Recursive TLV SDP parser**: replace byte-pattern matching with proper Bluetooth SDP TLV parser (~200 LOC rewrite). Works for current descriptor shape; fragile if Apple's descriptor grows past 127 bytes (length encoding shifts).
2. **PREfast vcxproj fix**: add `<CodeAnalysisRuleSet>NativeRecommendedRules.ruleset</CodeAnalysisRuleSet>` so static analysis becomes a CI gate again.
3. **Driver Verifier 0x49bb cycle test**: post-install validation of IRP completion paths under DV — requires the driver to be installed.
4. **VID format note**: INF uses `VID&0001004C` (Apple BT SIG company ID 0x004C with BT transport prefix). One reviewer mandated USB-VID-style `00010005AC` — that would not bind on BTHENUM and is wrong; current INF is correct.
